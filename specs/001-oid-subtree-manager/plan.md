# Implementation Plan: OID Universe — Federated Subtree Manager

**Branch**: `001-oid-subtree-manager` | **Date**: 2026-04-20  
**Spec**: [specs/plan.md](../plan.md) · [constitution](../constitution.md) · [technical-stack](../technical-stack.md)

## Summary

Self-hosted, federated OID subtree manager. The administrator installs the tool
with a configured `ROOT_OID` and manages the entire subtree below it.
Nodes are either `managed` (locally owned) or `federated` (read-only pointers
to external instances covering both upward context arcs and downward delegated
sub-subtrees). A single PostgreSQL table with `ltree` paths holds the full tree;
write-guard and visibility-cascade triggers enforce the constitution's invariants
at the DB layer. A FastAPI service exposes the API; a Next.js 14 admin UI (two
layouts: Explorer and Registry) provides the admin surface.

## Constitution Check

| Principle | Gate | Status |
|---|---|---|
| I. Federated Subtree Ownership | `ROOT_OID` from env; DB trigger rejects writes above root and on federated descendants | ✅ |
| II. Administrator-Centric Governance | All write paths require `X-Admin-Key`; Bearer tokens grant read-only | ✅ |
| III. Two-Tier Visibility + Inheritance | Explicit visibility required on `POST /oid`; cascade trigger fires on `private` update | ✅ |
| IV. Unified Federation Nodes | Single `node_type` enum; write guard returns `federation_url` on block; `delegate`/`reclaim` audited | ✅ |
| V. Subtree Integrity | `UNIQUE` on `oid_path`; `audit_log` append-only (no UPDATE/DELETE grants) | ✅ |
| VI. Simplicity | No live federation resolution, no cross-instance search, no contributor roles in v1 | ✅ |

All gates pass. No Complexity Tracking violations.
